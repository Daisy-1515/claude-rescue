## ✨ 特性

🎯 **零依赖** - 仅需 Node.js 16+，无需 `npm install`  
⚡ **实时监控** - 自动刷新，2 秒间隔实时显示会话状态  
🔍 **异常检测** - 自动识别重试、错误、无响应三种异常状态  
🖥️ **跨平台** - macOS / Windows / Linux 行为完全一致  
🛡️ **只读安全** - 仅读取会话文件，不修改或删除任何内容  
🚀 **一键接管** - 按 `o` 键开启接管向导，新开终端启动救援会话

## 📖 工作原理

每个运行中的 Claude Code 会话都会在 `~/.claude/sessions/<pid>.json` 写入状态文件。本工具读取该目录，展示交互式 TUI 界面，列出所有会话的：

- **sessionId** - 会话唯一标识
- **name** - 会话名称
- **directory** - 工作目录
- **status** - 状态（忙碌/空闲）
- **last activity** - 最后活动时间

> ⚠️ **重要**：该目录只包含**正在运行**的会话 — 会话退出后文件即被移除。因此这是一个**实时查看器**，只显示当前活跃的会话。
>
> 🔒 **安全保证**：本工具是**只读**的，只读取会话文件，从不修改或删除任何内容，可以放心地长期运行。

## 🚀 快速开始

### 环境要求

- **Node.js 16+**（检查版本：`node --version`）
- 仅此一项要求 — **无需 `npm install`**！

### 运行方式

**方式一：直接运行**

```bash
node sessions.js
```

> 💡 **提示**：建议在**独立终端窗口**运行（不要在 Claude 会话内），这样工具可以占用整个终端显示，你能看到所有活跃会话。

**方式二：全局安装**

```bash
npm install -g .
claude-rescue
```

安装后会自动生成跨平台启动器（Windows 上是 `.cmd`）。

## 📁 会话目录定位

工具会自动定位会话目录，所有平台逻辑完全一致：

| 条件 | 目录位置 |
| --- | --- |
| 设置了 `CLAUDE_CONFIG_DIR` 环境变量 | `$CLAUDE_CONFIG_DIR/sessions` |
| 否则（默认） | `~/.claude/sessions` |

其中 `~` 表示用户主目录：
- **Windows**：`%USERPROFILE%\.claude\sessions`（例如 `C:\Users\你的用户名\.claude\sessions`）
- **macOS/Linux**：`$HOME/.claude/sessions`

临时指定其他目录：`node sessions.js --dir <路径>`

## ⚡ 自动刷新

- 默认每 **2 秒**自动重读目录，保持列表和状态实时更新
- 按 **`a`** 切换自动刷新开关
- 按 **`g`** 手动刷新一次
- 第二行显示当前状态：`自动刷新: 2秒` / `自动刷新已关`

## 🔍 异常监控

工具会读取每个会话的 **transcript 尾部**（`~/.claude/projects/<编码目录>/<sessionId>.jsonl`），跳过元数据记录，只分析实质对话，将会话健康分为三种异常状态：

| 状态 | 颜色 | 判定条件 | 示例详情 |
| --- | --- | --- | --- |
| **重试中** | 🟡 黄色 | 最后一条实质记录是 API 重试（`system/api_error`） | `重试 3/10 · 520` |
| **错误** | 🔴 红色 | 最近 3 条实质记录中有 API 错误消息（`isApiErrorMessage`） | `524 · Error 524: A timeout occurred` |
| **无响应** | 🟡 黄色 | 进程存活，但会话已超过 **5 分钟**未更新 | `无响应 8 分钟` |

**显示方式：**
- 三种状态都在**状态列**显示对应颜色，前导圆点同步变色
- 标题栏显示异常计数（例如 `· 2 异常`）
- **详情视图**显示具体原因（状态码、重试次数、无响应时长等）
- 过滤器（`/`）支持输入 `异常`、`重试`、`错误`、`无响应` 来筛选
- `--json` 输出包含 `abnormal`（布尔）、`abnormalKind`（`ok` / `retrying` / `error` / `slow`）、`abnormalReason` 字段，便于脚本监控

**性能优化：**  
工具只读取 transcript **小尾部**（最大 256KB）："重试/错误"状态通过文件 mtime 缓存，"无响应"基于时间实时判断。即使 transcript 很大，检测速度依然很快。

> 📌 **注意**：检测目标是**当前运行**的会话（在 `sessions/` 目录中）。已报错/正在重试/卡住但进程仍存活的会话会被标记；一旦进程退出，会话不再在列表中（那是历史会话）。

## 🛟 接管卡住的会话

在列表或详情视图中，对选中的会话按 **`o`** 进入**接管向导**：

### 三步向导

1. **确认接管**  
   `y` 继续，其他键取消（避免误触）

2. **是否启用 `--dangerously-skip-permissions`？**  
   **默认否**：按 `n` / `enter` 跳过，按 `y` 启用

3. **是否命名（`--name`）？**  
   **默认否**：按 `n` / `enter` 跳过  
   按 `y` 后输入名字，再按 `enter` 确认（输入为空视为不命名）

### 执行流程

向导完成后，工具将：

1. **打开新终端**，位置为会话的**工作目录（cwd）**
   - macOS：Terminal.app（通过 `osascript`）
   - Windows：PowerShell（`start powershell`）
   - Linux：通用终端

2. **启动全新 Claude 会话**（**不使用 `--resume`**）  
   执行命令：`claude [--dangerously-skip-permissions] [--name <名字>] '<指令>'`  
   你选择的开关会附加在 `claude` 之后，指令之前

3. **指令内容**（中文）：  
   **「sessionId为xxx的任务卡住了，你帮我看看任务进度现在到哪里了，下一步我该做什么，请你继续执行任务。」**  
   其中 `xxx` 是卡住会话的 sessionId

这样，一个全新的 Claude 会话在同一项目目录启动，被告知哪个 sessionId 卡住了，它会检查进度、决定下一步、并继续任务 — 典型用于救援被标记为异常的会话。

**为什么不用 `--resume`？**  
因为新会话从（几乎）空上下文开始，请求很小，不会遇到导致旧会话长上下文持续重试的同样的速率限制/超时循环。

**操作提示：**
- 任何向导步骤按 **`esc`** 取消整个流程
- 命令（含你选择的开关）也会**复制到剪贴板**，终端未打开时可手动粘贴执行
- macOS 使用 Terminal.app；Windows 使用 PowerShell；其他系统尝试通用终端

## 🤖 非交互式使用（脚本 / 管道）

当输出不是终端，或传入参数时，工具打印结果后直接退出，不打开 UI：

```bash
node sessions.js --list      # 对齐的纯文本表格（中文标签）
node sessions.js --json      # JSON 数组（英文字段名，供脚本使用）
node sessions.js --json | jq -r '.[] | select(.alive) | .sessionId'
```

### 命令行选项

| 选项 | 说明 |
| --- | --- |
| `-l`, `--list` | 打印纯文本表格后退出 |
| `--json` | 输出 JSON（供脚本使用） |
| `--dir <路径>` | 使用指定的会话目录 |
| `--no-color` | 关闭颜色（也遵守 `$NO_COLOR` 环境变量） |
| `-h`, `--help` | 显示帮助信息 |
| `-v`, `--version` | 显示版本号 |

## ❓ 为什么没有"重命名 / 删除 / 清理"功能？

- **重命名**：会话的名字由**运行中的 Claude 进程**持有（保存在其内存和 transcript 中）。从外部编辑 `sessions/<pid>.json` 会被运行中的进程**立即覆盖**。**要重命名，请在 Claude 内使用内置命令 `/rename <名字>`** — 这是唯一可靠的方式。

- **删除 / 清理**：该目录只保存运行中的会话；删除文件后进程会很快重新写入。而且不存在需要清理的"已退出会话"（退出会自动移除）。

因此本工具专注于一件事 — **监控**。

---

## 📝 注意事项与已知限制

- **进程存活检测**使用 `process.kill(pid, 0)` — 它只探测进程是否存在，**不会实际发送信号或终止**进程。操作系统可能复用 PID，因此在极少数情况下判断可能不准确；这是尽力而为的检测。

- **Windows 终端兼容性**：颜色和全屏 UI 需要支持 VT 的终端（Windows Terminal，或 Windows 10+ 的 `conhost`，今天的默认终端都支持）。在很旧的控制台上，请改用 `--list` / `--json`。
